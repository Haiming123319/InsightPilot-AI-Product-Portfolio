# Phase 0/1 产品决策与技术改动记录

这份记录是作品集的决策证据。每项技术改动都必须能回答：解决谁的什么问题、带来什么产品价值、有什么风险、怎么评测、最终如何展示。

## 1. 基线冻结与代码审计

技术改动：新增 `docs/pre_api_baseline.md`、`docs/current_code_audit.md`，并记录 16 passed 的 API 接入前测试基线。

- 用户问题：用户无法判断当前 Demo 哪些能力真实可用，面试官也无法区分规则模拟和真实模型能力。
- 产品价值：建立可信的版本起点，后续可以把模型接入带来的变化与稳定性回归分开说明。
- 风险：只记录测试通过数会掩盖未覆盖的真实用户问题。
- 评测方式：固定记录测试总数、失败数、跳过数和耗时；补充真实调用链审计与未接入能力清单。
- 作品集呈现：用“基线冻结 + 能力审计”一页展示 MVP 边界和为什么没有直接宣称 AI 能力。

## 2. 统一解析结果与调用元数据

技术改动：新增 `ParserMetadata` 与 `IntentParseResult`，统一 Provider、模型、延迟、Token、request_id、回退和校验错误。

- 用户问题：用户不知道当前结果由规则还是模型产生，也无法判断等待时间和失败是否值得重试。
- 产品价值：让 Provider A/B 比较、失败解释和成本/体验权衡具备可观察数据。
- 风险：Token 或耗时字段可能缺失，展示不完整数据会造成伪精确。
- 评测方式：Mock usage、request_id 和 latency，验证成功、失败、回退三种状态都返回统一结构；缺失 Token 时显示“未返回”。
- 作品集呈现：放一张解析模式对比截图和字段说明表，强调“可观测”而不是只展示模型名称。

## 3. Provider Router 与失败回退

技术改动：新增 `src/services/intent_parser_router.py`，支持 `rule_based`、`openai`、`openai_with_fallback`。

- 用户问题：外部 API 超时、限流、鉴权失败时，用户不能因为一次解析失败而丢失分析机会。
- 产品价值：把模型能力作为可替换组件接入，规则基线承担可用性兜底，同时保留纯 OpenAI 模式用于对照实验。
- 风险：回退可能让用户误以为仍使用了模型；模型失败原因也可能被吞掉。
- 评测方式：分别注入超时、限流、鉴权和网络异常，验证纯 OpenAI 报错、回退模式继续运行，并在元数据和 Bad Case 日志中保留原因。
- 作品集呈现：展示“OpenAI 失败 -> 规则回退 -> 结果继续可用”的时序图和页面状态。

## 4. 受限数据画像与脱敏输入

技术改动：扩展 `DataProfile` 的语义类型、日期范围和 LLM payload；文本样例脱敏，样例数量和长度受限。

- 用户问题：模型需要知道字段是否可用、数据覆盖哪些周期，但用户不应为一次意图理解上传完整明细或敏感信息。
- 产品价值：提升模型字段选择和时间判断的上下文质量，同时降低隐私、Token 和成本风险。
- 风险：过度脱敏会损失业务语义，过度暴露又可能泄露个人或项目明细。
- 评测方式：断言请求中不出现完整 DataFrame；验证文本样例为 `[redacted]`、样例数量不超过上限，且只发送数据画像字段。
- 作品集呈现：用“发送给模型 / 留在本地”的边界图，清楚展示数据最小化决策。

## 5. Pydantic 结构化输出与本地合法性校验

技术改动：OpenAI 优先使用 Responses API 结构化输出；新增 `intent_validation.py` 校验字段、操作、图表、类型和同比时间范围。

- 用户问题：模型可能输出不存在字段、未支持操作或超出数据范围的时间条件，导致用户得到错误结果或页面崩溃。
- 产品价值：把模型的不确定性转化为可解释的澄清，而不是让错误意图进入分析执行。
- 风险：校验过严会增加澄清率，用户可能觉得系统“不够聪明”。
- 评测方式：构造不存在字段、未知操作、非法图表、单周期同比和越界时间用例，要求 `clarification_needed=true` 且执行按钮禁用。
- 作品集呈现：展示一个“字段幻觉拦截” Bad Case，并用职责分工图说明模型、Python、规则和用户的边界。

## 6. Streamlit 解析模式与可观测状态

技术改动：侧边栏增加三种解析模式；结果区展示实际 Provider、模型、耗时、Token、回退和校验错误。

- 用户问题：用户无法理解结果来源，也无法在模型失败后自主选择是否切换规则模式。
- 产品价值：提高信任和可恢复性，让产品演示从“黑盒问答”变成“可比较的分析工作流”。
- 风险：指标太多会干扰主任务；真实 API 未运行时不能伪造 Token 或效果提升。
- 评测方式：页面冒烟检查三种模式的状态文案；无 Key 时验证页面保持可操作，并显示切换建议。
- 作品集呈现：截图只保留关键状态字段，并在旁边标注哪些指标来自 Mock、哪些需要真实 API 运行后补充。

## 7. Phase 1 失败路径测试与报告

技术改动：新增 `tests/test_intent_parser_router.py` 和 `docs/phase1_real_intent_parser_report.md`，测试数从 16 增至 29。

- 用户问题：只测成功路径无法证明 AI 产品在真实网络和模型不确定性下可用。
- 产品价值：把风险控制变成可复现验收标准，并为后续真实模型评测留下对照组。
- 风险：Mock 通过不等于真实 API 通过，不能把结构正确性误写成模型效果。
- 评测方式：29 个自动化测试；真实 API 任务理解、成本、P95 延迟仍明确标记为未测量。
- 作品集呈现：用测试矩阵展示“已验证的稳定性”和“下一阶段待测的模型效果”，体现范围控制能力。
