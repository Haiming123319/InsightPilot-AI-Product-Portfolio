# InsightPilot 后续产品路线图

本文件回答一个问题：在当前“单文件费用分析 + Python 计算 + 自动校验 + 评测”的基础上，如何继续把 InsightPilot 做成更完整、也更适合 AI 产品经理作品集展示的产品。

## 1. 先确定产品主线

下一阶段不追求功能数量，而是继续强化 InsightPilot 的差异化：

> 让业务用户可以修改分析过程，并且知道每个结论是如何计算、如何验证、如何导出的。

因此后续版本优先补齐四个闭环：

1. 可编辑：用户能修改字段映射、清洗规则和计划步骤。
2. 可追问：用户能基于上一次结果继续分析，例如排除异常值后重算。
3. 可观测：能看到模型、工具、校验器的耗时、Token 和失败原因。
4. 可评测：能比较不同模型和不同防护层对任务完成率、数值准确率和成本的影响。

多文件、数据库、权限和协作属于产品扩展面；多 Agent、RAG、微调和 PPT 属于技术或交付增强面，不应在下一版同时进入核心链路。

## 2. 版本路线

| 版本 | 目标 | 作品集中的证明材料 | 优先级 |
|---|---|---|---|
| V6 | 可编辑、可追问、可观测的增强版 | 操作截图、事件日志、追问前后对比 | P0 |
| V7 | 多文件关联和真实模型对比实验 | Join 计划、三方案评测、成本表 | P1 |
| V8 | 数据库、权限和团队协作的企业化设计 | 架构图、权限矩阵、审计日志原型 | P2 |
| V9 | 报告交付增强 | PPT 生成样例、引用检查、模板配置 | P2 |

## 3. V6：优先做成可运行增强版

### 3.1 用户可编辑字段映射

当前系统默认使用 `date`、`department`、`expense_type` 和 `amount`。下一版增加一个“字段映射”确认区：

- 原始字段下拉选择：日期、金额、部门、费用类型、项目、地区。
- 展示类型转换预览和受影响行数。
- 用户可以把业务字段映射到标准字段，也可以明确标记“暂不使用”。
- 保存一份 `mapping_config`，后续计划和工具只读取标准字段，不直接依赖原始列名。

建议数据结构：

```json
{
  "date": "发生日期",
  "amount": "含税金额",
  "department": "成本中心",
  "expense_type": "费用科目",
  "ignored_columns": ["备注"]
}
```

验收标准：用户修改字段后，数据体检、分析计划、计算结果和溯源中的字段名称保持一致；缺少必要字段时进入澄清，而不是静默计算。

### 3.2 用户可编辑清洗规则

先支持有限且可解释的规则，不做任意代码编辑：

- 缺失金额：排除 / 按 0 处理 / 要求用户补齐。
- 重复记录：保留第一条 / 保留最后一条 / 全部保留并提示。
- 部门名称：去除空格、统一别名。
- 异常值：纳入分析 / 排除分析 / 单独标记。

每条规则都记录 `before_count`、`after_count` 和 `rule_id`，报告中展示“原始行数 → 纳入计算行数”的变化。

### 3.3 用户可编辑分析计划

计划步骤改为可操作列表，支持：

- 勾选启用或停用步骤。
- 修改步骤顺序。
- 修改有限的参数，例如同比年份、异常阈值方法、Top N。
- 执行前重新校验依赖关系。

不允许用户直接输入 Python 代码。产品要保护“用户可控制”和“系统可验证”之间的边界。

验收标准：计划被修改后，执行日志、结果、公式和校验报告必须使用修改后的计划；依赖缺失时阻止执行并给出可读提示。

### 3.4 多轮追问

先实现一个明确场景：

> “排除异常值后重新分析”

系统保留上一次的 `dataset_id`、`mapping_config`、`cleaning_config`、`intent` 和 `result_id`，追问只生成差异配置，不重新猜测全部上下文。

后续可扩展：

- “只看华东地区”。
- “把同比改成环比”。
- “按项目再下钻一层”。
- “把结论改成给管理层看的三句话”。

### 3.5 执行日志和真实模型适配

增加统一事件记录：

```text
task_started
intent_parsed
plan_generated
tool_started
tool_finished
validation_finished
result_exported
task_failed
```

每条事件至少包含：`task_id`、`step_id`、`component`、`model`、`latency_ms`、`input_tokens`、`output_tokens`、`status`、`error_type`。

`llm_service.py` 保留当前规则版实现，同时增加统一接口：

```text
IntentParser
├── RuleBasedIntentParser
└── OpenAIIntentParser
```

这样可以在不改动任务执行和校验层的前提下切换模型，并为三方案实验提供同一套输入输出协议。

## 4. V7：多文件关联与三方案评测

### 4.1 多文件关联

推荐场景是“费用明细 + 部门主数据”，而不是一开始做任意 SQL Join：

- 用户上传主表和明细表。
- 系统识别候选关联键，例如 `department_id`。
- 展示关联覆盖率、未匹配行数和一对多风险。
- 用户确认 Join 类型后，生成可审计的关联步骤。

关联结果要保存：文件名、表名、关联键、Join 类型、匹配率和未匹配样例。关联失败时保留原始数据，不直接丢行。

### 4.2 三方案对比实验

固定同一批测试问题、同一份数据和同一评分标准，比较：

| 方案 | 负责内容 | 需要观察的问题 |
|---|---|---|
| 纯 LLM | 直接生成答案 | 数值幻觉、格式不稳定、不可复核 |
| LLM + Python | 模型规划，Python 计算 | 计算可靠性提升，但计划错误仍可能传递 |
| LLM + Python + 校验 | 规划、计算、发布前校验 | 准确性、拒答率、延迟和成本的平衡 |

至少记录：任务完成率、意图准确率、数值准确率、结论一致率、校验拦截率、幻觉率、平均延迟、Token 和单任务成本。

三方案实验的价值不是证明“加层越多越好”，而是展示产品取舍：校验层可能增加延迟和拒答，但能降低不可追溯的错误结果。

## 5. V8：企业化能力的合理边界

### 5.1 数据库连接

先支持只读连接和白名单查询，不允许模型直接拼接并执行任意 SQL：

1. 用户选择数据源和允许访问的表。
2. 系统读取 schema，不读取全库数据。
3. 模型生成结构化查询计划。
4. SQL 生成器根据白名单渲染查询。
5. 执行前做权限、行数、成本和敏感字段检查。
6. 查询结果进入与文件分析相同的标准数据契约。

作品集阶段可以先做连接配置页和安全架构图，真实数据库接入可作为后续工程版本。

### 5.2 企业权限管理

最小可行角色：

| 角色 | 数据 | 能力 |
|---|---|---|
| 查看者 | 仅可见已发布结果 | 查看、导出 |
| 分析者 | 可访问授权数据集 | 上传、分析、追问 |
| 管理者 | 管理数据源和成员 | 授权、撤销、查看审计 |

所有数据集、任务和导出物都绑定 `workspace_id`。权限判断放在服务层，不依赖 Streamlit 页面是否显示按钮。审计日志记录谁在什么时间访问了什么数据、执行了什么任务、导出了什么结果。

### 5.3 团队协作

先做“可共享分析资产”，暂不做实时多人协同编辑：

- 分析任务可发布为只读链接。
- 结果、计划、数据版本和校验报告一起保存。
- 支持评论和标记“需复核”。
- 用版本号区分原始结果、修改后结果和发布结果。

这已经足以体现协作价值，也不会引入实时协同、冲突合并等过重基础设施。

## 6. 多 Agent、RAG、微调和 PPT 的判断

### 多 Agent

暂不拆成多个自由对话 Agent。当前更适合使用一个编排器加多个职责明确的工具：解析器、计划器、Python 执行器、校验器、解释器和导出器。只有当任务类型、数据源和权限规则明显增加后，才考虑按领域拆分 Agent。

### RAG

RAG 不应被用来弥补表格计算能力。它适合补充企业指标口径、费用制度、部门层级和历史分析报告。下一版可增加“指标口径知识库”，并要求每条引用带来源、版本和生效日期；如果没有相关文档，系统必须明确说明未检索到依据。

### 模型微调

目前不建议微调。规则版基线和真实模型对比还没有完成，样本量也不足。先收集意图解析、字段映射、澄清和计划修改的脱敏标注数据，只有当 Prompt、结构化输出约束和 RAG 仍无法解决稳定性问题时，再评估微调收益。

### 自动生成 PPT

PPT 是交付层，不是分析正确性的核心。应在结果校验通过后生成，内容来自已验证的结论、图表和证据，而不是让模型直接写一份新的数字。第一版支持固定模板：标题页、数据质量、关键结论、趋势图、异常记录、方法与限制、附录。

## 7. 推荐实施顺序

### 第一个迭代周期：做 V6

- 字段映射编辑
- 清洗规则编辑
- 计划步骤编辑
- 排除异常值后追问
- 执行事件日志
- 真实模型适配器接口

完成后，作品集的核心叙事从“我做了一个 Demo”升级为“我设计了一个可控、可追溯的 AI 分析工作流”。

### 第二个迭代周期：做 V7

- 多文件关联
- 30-50 条扩展测试集
- 三方案对比实验
- Token、成本和延迟统计
- 5-8 个 Bad Case 复盘

完成后，可以用数据证明架构选择，而不是只展示功能截图。

### 第三个迭代周期：做 V8/V9 设计验证

- 只读数据库连接原型
- 权限矩阵和审计日志
- 可共享分析资产
- 指标口径 RAG 原型
- 基于校验结果的固定模板 PPT

这些能力可以先用原型、接口契约和架构图表达，等核心分析链路稳定后再工程化。

## 8. 最终验收门槛

下一版不以“完成了多少功能”作为唯一标准，而以以下结果作为发布门槛：

- 用户能修改字段映射并得到一致的分析结果。
- 用户能修改计划参数，并在日志中看到差异。
- 用户能通过至少一个追问复用上下文。
- 所有发布结论都有证据、公式和校验状态。
- 三方案实验至少覆盖 30 条问题，并公开记录成本、延迟和错误。
- 失败任务有清晰的澄清、拒答或重试路径。
- README、PRD、评测结果、验证报告和作品集内容使用同一组最新数字。

## 9. 面试中的表达方式

可以把下一步总结为：

> 我没有把所有企业级 AI 名词都堆进 MVP，而是先围绕“可控、可验证、可追溯”补齐用户编辑、上下文追问和评测体系。多文件和数据库是数据接入扩展，权限和协作是企业化扩展，RAG、微调和多 Agent 则根据真实误差和规模再决定是否引入。这样每个技术选择都有对应的问题、指标和验收标准。
